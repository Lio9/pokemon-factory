# Pokemon Factory 优化计划

最后更新：2026-04-13
状态：草案，可直接修改

## 1. 文档用途

这份文档用于整理 Pokemon Factory 当前阶段的优化方向，目的是：

- 统一短期和中期改造目标
- 给后续开发排出优先级，而不是并行做大量分散优化
- 明确哪些项可以立即开始，哪些项需要先做设计
- 作为后续 issue / milestone 的母文档

使用建议：

- 任务开始前，先确认本周只推进 1 到 2 个重点主题
- 每完成一项，把状态从 `待做` 改成 `进行中` 或 `完成`
- 如果方案调整，优先修改“阶段目标”和“任务拆分”，不要只改结论

## 1.1 当前硬约束

这份计划默认遵守以下约束：

- 数据库保持当前嵌入式方案，不在本计划中调整数据库引擎
- 不做数据库迁移、表结构重写、初始化链路重写、存储方案替换
- 后续如果需要转向 Android，优先保证业务层、接口契约、资源组织方式可迁移
- 当前优化范围聚焦在业务结构、前端状态、测试、鉴权、日志、运维基线和客户端适配准备

## 2. 当前判断

项目当前已经具备可运行的前后端链路，但主要问题不在“功能缺失”，而在“维护成本正在快速上升”。

当前最明显的几个热点：

1. battleFactory 的战斗核心过大，`BattleEngine` 已经承担过多职责。
2. pokeDex / common 的导入、初始化、伤害计算逻辑都偏重，实现类过大。
3. 前端大页面较多，状态、请求、UI 交织，后续继续扩展会越来越难维护。
4. 鉴权、错误响应、日志、部署等横切能力还不够统一。
5. 测试能力分布不均，battleFactory 有一定基础，其他区域相对薄弱。

结论：

- 现在不建议优先做“大迁移”，比如直接拆微服务或直接更换数据库。
- 数据库相关方案保持不动，本计划不包含数据库引擎、存储方式和迁移策略调整。
- 更合理的路径是：先做模块化单体优化，把边界、状态、构建、测试、运维基线和 Android 迁移准备补齐。

## 3. 优先级原则

本计划采用以下优先级原则：

### P0：立即做

- 能明显降低维护成本
- 风险低或可控
- 会影响后续所有开发效率

### P1：尽快做

- 不是当前阻塞项，但已经明显开始拖累项目
- 需要少量设计后推进

### P2：后续评估

- 收益存在，但必须建立在前两层完成后
- 容易演变成高成本改造

## 4. 重点问题清单

### 4.1 后端核心复杂度过高

现象：

- `battleFactory` 的战斗逻辑集中在超大类中
- `common` 和 `pokeDex` 中也存在超大服务类与初始化类
- 同一类里同时处理状态、规则、持久化编排、错误兜底

影响：

- 修改成本高
- 定位回归问题困难
- 单元测试边界不清晰
- 新特性更容易引入连带 bug

建议：

- 优先拆 battle 引擎
- 再拆服务编排与计算链路
- 拆分时先按职责切，不先追求抽象层次“优雅”

### 4.2 模块边界还不够清晰

现象：

- `common` 承担了较多共享能力和初始化入口
- `battleFactory` 直接依赖 `common`、`user-module`
- 当前运行强依赖启动顺序

影响：

- 模块复用边界容易失控
- 小改动可能触发多模块连锁影响
- 本地启动和部署方式不够稳定

建议：

- 先把“公共基础设施”和“业务实现”边界整理清楚
- 优先消除模块职责混杂，再决定是否需要更大架构调整
- 边界整理只针对业务与调用关系，不调整数据库层方案

### 4.3 前端页面和数据层耦合偏重

现象：

- 大页面承担过多流程控制职责
- composable 中混有真实请求、UI 兜底和示例数据
- 认证状态和 API 调用仍然是半集中、半散落的方式

影响：

- 页面继续扩展会越来越难改
- 同类逻辑难复用
- 排查状态问题成本高

建议：

- 先拆 battle 页面的编排逻辑
- 明确 store / composable / view 的职责边界
- 让示例数据与真实接口彻底分离

### 4.4 横切能力缺少统一基线

现象：

- 错误响应结构还不完全统一
- 安全配置存在分散实现
- 结构化日志、健康检查、容器化基础不足

影响：

- 前后端联调体验不稳定
- 上线和排障成本偏高
- 后续规模增长后会越来越痛

建议：

- 统一错误码和响应结构
- 收口 JWT 相关能力
- 补齐 Actuator、日志规范、最小容器化方案

### 4.5 测试分布不均衡

现象：

- `battleFactory` 已有一定测试基础
- `pokeDex`、`common`、前端测试明显不足

影响：

- 结构性改造时信心不足
- 只能依赖人工回归

建议：

- 不追求一次补全所有测试
- 优先给高风险路径补 smoke / integration / contract test

### 4.6 Android 迁移准备不足

现象：

- 当前前端仍然强依赖浏览器环境和 Web 页面编排方式
- 平台相关能力和业务能力还没有明确边界
- 未来如果转 Android，容易把页面逻辑、状态逻辑和平台能力耦合搬过去

影响：

- 后续 Android 迁移成本可能偏高
- 复用边界不清晰，容易出现“业务逻辑跟着 UI 走”
- 平台切换时验证范围会过大

建议：

- 提前收口前端状态层、接口层和平台依赖点
- 尽量让 battle、auth、pokemon 数据流保持平台无关
- 把 Android 迁移准备聚焦在客户端结构，不触碰数据库方案

## 5. 三阶段优化路线图

### 阶段一：低风险高收益优化

时间建议：1 到 2 周
目标：先降低维护噪音，给后续结构性拆分扫清障碍。

#### 阶段一目标

- 统一关键接口行为
- 收口认证与状态处理
- 补齐基础运维能力
- 明确嵌入式与 Android 迁移约束
- 为后续重构建立更稳定的地基

#### 阶段一任务

- [x] 统一后端 API 响应结构与错误码
- [x] 统一 battle / user 的异常输出格式
- [x] 整理前端 `api.js` 的错误解析策略，减少页面级兼容逻辑
- [x] 收口前端登录态和 battle 运行态的状态管理边界
- [x] 清理 `usePokemonData` 中的示例数据兜底逻辑
- [x] 为 `battleFactory` 补齐 stateless 安全配置一致性
- [x] 在项目文档中固定“数据库不动、嵌入式优先、为 Android 预留结构”的约束
- [x] 增加基础健康检查接口
- [x] 增加结构化日志格式
- [x] 补最小 Dockerfile 与本地 docker-compose 草案

#### 阶段一收益

- 联调更稳定
- 调试和排障更快
- 后续重构风险下降

#### 阶段一风险

- 低
- 主要风险来自接口返回格式调整，需要前端同步验证

---

### 阶段二：结构性拆分

时间建议：3 到 5 周
目标：拆掉当前最主要的复杂度热点，但保持整体架构仍然可控。

#### 阶段二目标

- 拆分战斗核心职责
- 拆分服务编排和计算链路
- 清理 common 的业务职责边界
- 让大页面和大类不再继续膨胀

#### 阶段二任务

- [ ] 将 `BattleEngine` 拆为多个职责明确的组件（已完成状态访问、预览选择、场地效果、决策编排、伤害计算、状态结算、AI 分析、AI 选招、AI 换人、行动构建、目标解析、单 action 执行、回合末清理、残局/补位编排，以及 preview/setup/replacement/exchange façade helper 抽离；现有 battleFactory 模块链测试 93/93 通过）
- [x] 继续拆分开场初始化、预览确认与补位提交路径中的 façade 编排壳层
- [ ] 评估 action 排序与共享 rule helper 是否值得继续抽离，避免在收益递减区继续制造过细碎片
- [ ] 为 battle 拆分后补齐对应测试
- [x] 将 `Battle.vue` 拆为页面编排层 + 业务区块组件
- [x] 将大 composable 再拆为更细粒度的数据访问和状态控制单元
- [x] 重新整理 `common` 中的共享基础设施与业务逻辑边界，但不调整数据库层（已完成图鉴 service、vo、mapper、model 与 damage util 迁入 pokeDex，并已收紧 pokeDex 的 mapper 扫描范围；后端 reactor `mvn clean test` 已通过）
- [x] 抽离 battle / auth / pokemon 相关的稳定 DTO 和状态转换边界，方便未来 Android 复用

#### 阶段二收益

- 修改影响面更小
- 新功能接入更容易
- 核心逻辑更容易测试和定位问题

#### 阶段二风险

- 中等
- 需要依赖现有测试和补充测试保障重构质量

---

### 阶段三：Android 迁移准备与平台适配

时间建议：4 到 6 周
目标：在结构稳定后，为未来 Android 迁移做客户端结构准备，同时保持嵌入式数据库方案不变。

#### 阶段三目标

- 提炼可迁移到 Android 的业务与状态边界
- 降低对 Web 专属能力的直接依赖
- 提升客户端适配能力和发布准备度

#### 阶段三任务

- [ ] 为关键接口建立性能基准数据
- [ ] 梳理前端对浏览器专属 API 的依赖点
- [ ] 输出 Android 迁移方式草案（例如 WebView / Hybrid / 原生壳集成）
- [ ] 规划前端 API 层与平台适配层边界
- [ ] 整理 battle / auth / pokemon 相关页面的移动端拆分方案
- [ ] 整理图片、静态资源、规则数据的本地打包策略
- [ ] 增加关键链路监控指标
- [ ] 规划部署流水线与发布规范
- [ ] 如有必要，再评估更进一步的模块解耦方案

#### 阶段三收益

- Android 迁移时可复用更多现有业务能力
- 平台切换范围更可控
- 在不改数据库的前提下提升后续扩展空间

#### 阶段三风险

- 中等
- 主要风险来自客户端边界抽象不当，需要避免为了 Android 过早过度设计

## 6. 明确暂不优先做的事情

以下事项暂不建议作为当前优先级最高的工作：

- [ ] 直接拆微服务
- [ ] 调整数据库引擎、迁移数据库或重写数据库初始化链路
- [ ] 先做大规模前端技术栈替换
- [ ] 在核心复杂度未下降前引入过多基础设施

原因：

- 这些工作成本高
- 回报不一定立刻出现
- 容易掩盖当前真正的结构问题

## 9. 风险与前提

开始执行前，建议先确认以下前提：

- 是否接受先做“模块化单体优化”，而不是立刻做大架构升级
- 是否允许在短期内优先做结构整理，而不是继续堆功能
- 是否愿意把部分前端开发时间投入到状态和页面拆分
- 是否愿意为重构补少量测试，而不是完全依赖手工验证

如果以上前提不成立，需要回头重排优先级。

## 10. 决策记录

### 决策 1

- 决策：当前阶段不优先拆微服务
- 原因：核心复杂度问题主要在模块内部，而不是服务数量不够
- 日期：2026-04-10

### 决策 2

- 决策：数据库保持当前嵌入式方案，当前阶段不做数据库相关改造
- 原因：后续需要保留向 Android 演进的空间，当前优化聚焦业务结构与客户端适配准备
- 日期：2026-04-10

### 决策 3

- 决策：先做 battle / 前端数据层 / 服务编排层的职责拆分
- 原因：这些区域是最主要的维护热点，且不需要调整数据库方案
- 日期：2026-04-10

### 决策 4

- 决策：后续如需转 Android，优先复用业务层、接口层和状态层，不先动数据库方案
- 原因：平台迁移的主要风险在客户端结构和边界，而不是当前数据库形态
- 日期：2026-04-10
