# Android 迁移约束草案

最后更新：2026-04-10
范围：整理 Android 迁移前置约束与客户端边界，不涉及数据库调整和最终迁移方案定案

## 1. 这份草案解决什么问题

当前项目已经明确“后续可以转 Android”，但如果只停留在口头约束，后面前端和接口层还是会继续往 Web 专属方向发散。

这份草案先把三个问题钉住：

- 哪些东西现在就必须保持平台无关
- 哪些东西暂时可以继续留在 Web 层
- Android 迁移时当前项目不打算动什么

## 2. 当前不会动的内容

Android 迁移准备阶段，以下内容不在范围内：

- 不更换数据库引擎
- 不迁移 SQLite 到其他数据库
- 不重写数据库初始化链路
- 不重写 battle summary 的后端存储方式
- 不先拆微服务来配合 Android

结论：

Android 迁移的主要工作面在客户端结构、接口契约和资源组织，不在数据库方案。

## 3. 当前必须保持平台无关的边界

### 3.1 battle 数据契约

当前已开始独立的内容：

- battle 响应归一化
- 工厂挑战 run 状态归一化
- battle move 提交 payload 组装
- battle 结算摘要归一化

要求：

- battle 相关响应结构必须继续走稳定 contract
- 不允许在页面组件里继续直接拼接后端字段兼容逻辑

### 3.2 auth 数据契约

当前已开始独立的内容：

- 登录 / 注册 / 恢复会话的 session 归一化

要求：

- 登录态处理继续停留在 contract + composable 层
- 不把 token / user 字段解析散落回页面

### 3.3 pokemon 数据契约

当前已开始独立的内容：

- 详情、技能、特性、进化链的统一解包和失败处理

要求：

- 图鉴数据继续只通过 contract / composable 进入页面
- 不再恢复 sample fallback 这类只对 Web 调试友好的隐式兜底

## 4. 当前允许继续留在 Web 层的内容

以下内容暂时可以继续保留在 Web 页面层，不需要为了 Android 过早抽象：

- 纯样式布局
- 断点响应式表现
- Element Plus 组件装配
- 移动端浮动操作条的视觉交互
- 弹窗和折叠面板这类纯展示容器

原则：

- UI 外壳可以是 Web 专属
- 但驱动 UI 的状态、接口和业务转换边界不能继续绑定浏览器实现

## 5. 当前 battle 页面对应的迁移准备现状

已经完成的准备：

- `Battle.vue` 只保留页面编排，业务区块拆成独立组件
- battle 页面派生状态已拆到独立 composable
- battle / auth / pokemon contract 已独立到 `src/services/contracts/`

这意味着：

- 页面组件未来可以被 Android 页面结构替换
- 只要 contract 和状态层保持稳定，客户端迁移时可以减少直接改业务逻辑的范围

## 6. Android 迁移前必须继续收口的点

在真正开始 Android 方案评估前，建议继续收口以下内容：

### 6.1 API 层

- 明确 battle / auth / pokemon 的 request builder 和 response contract
- 降低页面对原始响应字段的认知

### 6.2 状态层

- 继续避免“大 view + 大 composable + 大量局部兼容逻辑”回流
- 让状态控制尽量围绕流程而不是围绕具体组件

### 6.3 平台依赖点

- 标记浏览器专属能力
- 区分“业务需要”和“仅 Web 交互需要”

## 7. 后续方案评估时的判断标准

等后面真正评估 Android 落地方式时，不管是 WebView、Hybrid 还是原生壳集成，都应优先看这些标准：

- battle / auth / pokemon contract 是否已经稳定
- 状态层是否足够脱离页面细节
- 是否还存在大量浏览器专属逻辑混在业务流程里
- 静态资源、规则数据和图片是否已有清晰打包边界

## 8. 当前结论

Android 迁移准备这件事，现在最该做的不是“马上决定技术路线”，而是继续确保：

1. 数据契约稳定。
2. 状态层不回流进页面。
3. Web 专属 UI 和业务流程边界继续分开。

只要这三点守住，后面无论选哪种 Android 承载方式，成本都会比现在低得多。
